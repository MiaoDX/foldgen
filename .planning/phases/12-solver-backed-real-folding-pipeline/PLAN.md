# Phase 12: Solver-backed Real Folding Pipeline

## Goal

Turn the current schematic pipeline into a solver-backed target-result pipeline
for at least one case, using `simple-boat` as the forcing function.

The phase is successful only if completed-target output is backed by external
folded-state evidence. A prettier 2.5D canvas does not satisfy this phase.

## Background

The current implementation can generate FOLD files, crease SVGs, action
sequences, and deterministic preview animation. That is useful infrastructure,
but it is not enough to claim that a target has been folded.

Known failure:

- `simple-boat` is built from curated mast/hull/sail cues.
- Its Flat-Folder validation currently fails.
- The preview is generated by `packages/fold-core/src/preview.mjs`, a local
  heuristic that moves vertices for inspection.
- The demo can visually imply target completion even when no solver-backed
  folded state exists.

## Scope

1. Add a solver-state adapter around Flat-Folder.
2. Store solver-backed folded-state artifacts.
3. Make completed-target status require solver-backed state.
4. Replace or demote `simple-boat` so it cannot appear as a successful case
   unless it passes the new gate.
5. Preserve existing executor-readable diagrams, but mark them as dependent on
   the same selected fold state.
6. Keep the existing inspection preview only as a debug/inspection artifact.

## Non-goals

- No physical embodiment claim.
- No Rabbit Ear core dependency.
- No full Origami Simulator embed.
- No broad target generation research.
- No UI redesign beyond status/gate surfaces needed for the new evidence chain.

## Implementation Plan

### 1. Extract Flat-Folder solver state

Add a new function in `packages/foldgen-agent/src/community-adapters.mjs`, or a
small sibling module if cleaner:

- input: generated FOLD path
- output:
  - `flat-folder-state.json`
  - `folded-state.fold`
  - solver status
  - state count
  - folded vertex coordinates if available
  - face order / layer order if available
  - conflict details on failure

Use the same normalized solver input that `validateFlatFolderArtifact` uses, so
extra history edges that are not part of `faces_vertices` do not crash the
solver path.

### 2. Add a hard result gate

Update pipeline case classification:

- `status: valid` for completed-target cases requires:
  - local structural validation pass
  - community FOLD compatibility pass
  - Flat-Folder validation pass
  - solver-state artifact exists
- `result_quality.physical_result_status` stays `not-proven`.
- `result_quality.preview_status` must distinguish:
  - `solver-backed-folded-state`
  - `2.5d-inspection-only`
  - `missing`
- `result_quality.target_match_status` must not be `target-match-passed` until
  Phase 14 adds scoring.

### 3. Fix boat truthfulness

Use one of these outcomes:

- Preferred: replace `simple-boat` with a known-good public or repo-authored
  boat FOLD/sequence that passes Flat-Folder and produces a folded-state
  artifact.
- Acceptable for this phase: remove/demote `simple-boat` from completed targets
  and mark it as `solver-failed` with the conflict evidence.

Do not keep the current boat cues as a successful target.

### 4. Update artifacts and demo data

Add artifact paths:

- `flat_folder_state`
- `folded_state_fold`
- optional `solver_backed_preview` if a preview JSON is derived from the
  folded-state artifact

Update demo summary rendering so the evidence chain is explicit:

- Completed result: only when solver state exists.
- Inspection result: current preview can be shown, but not as final folded
  target evidence.
- Failed result: conflict/error message is visible.

### 5. Tests

Add tests that fail against the current implementation:

- A Flat-Folder failed case cannot have completed-target status.
- `simple-boat` cannot be `valid` unless a folded-state artifact exists.
- Solver-state adapter writes deterministic artifacts for a passing fixture.
- Demo summary includes solver-state status and does not call 2.5D preview a
  completed result.

## Gate

```bash
npm run m12:solver-state
npm test
npm run validate:stage1
```

If `npm run validate:flat-folder` still fails because some non-completed cases
are intentionally blocked, that is acceptable only when the phase-specific
summary records those cases as blocked rather than successful.

## Acceptance Criteria

- At least one target case has a solver-backed folded-state artifact.
- No case with failed Flat-Folder validation is displayed as completed target
  output.
- `simple-boat` either passes the solver-backed gate or is visibly demoted.
- The 2.5D preview is labeled and stored as inspection-only.
- The public claim remains `simulator-valid / executor-readable /
  embodiment-untested` only for cases that satisfy the updated simulator-valid
  definition.

## Follow-up

Phase 13 should replace the completed-result preview with Three.js/WebGL
rendering from `folded-state.fold`. Phase 14 should add target-match scoring so
solver-valid but visually wrong shapes are rejected.
